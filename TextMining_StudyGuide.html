01 — What is the project

Project Overview

Input

Raw text reviews from 10,000 hotel customers. Each entry contains a hotel name, a free-text review, and a 1–5 star rating.

Output

Per-hotel sentiment score (0–1) representing the proportion of positively classified reviews. A ranked list of hotels by guest satisfaction.

The project builds an end-to-end Natural Language Processing (NLP) pipeline: it takes messy, human-written text, transforms it into numbers a machine can learn from, trains a classifier, and then aggregates predictions into actionable hotel rankings.

Core Question the Model Answers

Given only the text of a review — without looking at the star rating — is this guest happy or unhappy?

02 — End-to-end flow

Full Pipeline Architecture

Every step transforms data into a form the next step can use. Nothing is skipped — the model never sees raw text directly.

01 Raw CSV 10,000 rows
02 Feature Select & Label 3 cols + binary y
03 Text Cleaning lower / punct / lemma
04 TF-IDF → (9999 × 5000) matrix
05 Train / Test Split 80% / 20%
06 Logistic Regression binary classifier
07 Evaluate accuracy, F1
08 Aggregate hotel rankings
Key insight about the pipeline

The star rating is used only during training to create the ground-truth label. At inference time the model predicts sentiment purely from text — that is the whole point. If we already had the rating, we wouldn't need the model.

03 — Text cleaning theory

Text Preprocessing

Raw text is full of noise. Preprocessing normalises it so that semantically identical words map to the same token and irrelevant characters do not inflate the feature space.

StepWhat it doesWhy it mattersExample
Lowercasing All text → lowercase "Great" and "great" must be the same token Great → great
Punctuation removal Strip non-alphanumeric chars Punctuation carries no sentiment signal for TF-IDF great!!! → great
Tokenization Split string → word list Enables per-word operations "great room" → ["great","room"]
Stopword removal Delete common function words Words like "the","is","and" appear everywhere — zero discriminative power ["the","great","room"] → ["great","room"]
Lemmatization Reduce to dictionary base form Consolidates inflected forms; improves generalisation running → run, rooms → room

Lemmatization vs Stemming — what's the difference?

Lemmatization (used here)

Uses a vocabulary and morphological analysis to return the proper dictionary form (lemma). "better" → "good". Linguistically correct.

Stemming (NOT used)

Crude suffix-chopping. "running" → "runn". Faster but produces non-words. Can hurt accuracy on sentiment tasks.

Sentiment distribution after labelling

Out of 9,999 clean records: 7,285 positive (≥4 stars) and 2,714 negative (<4 stars). This ~73/27 split is a class imbalance that directly impacts model performance — discussed in Limitations.

04 — Feature extraction theory

TF-IDF Vectorization

The model cannot read text. TF-IDF converts each review into a row of numbers — one number per vocabulary word — encoding how important each word is to that review.

What is TF-IDF?

TF-IDF stands for Term Frequency × Inverse Document Frequency. It rewards words that are common in one document but rare across the whole corpus.

TF(t, d) = count of term t in document d / total terms in d How often does this word appear in this review?
IDF(t) = log( N / df(t) ) + 1 N = total documents; df(t) = number of docs containing term t. Words appearing in every doc get low IDF.
TF-IDF(t, d) = TF(t, d) × IDF(t) High score = word is frequent HERE but rare elsewhere → highly discriminative signal.

Concrete example

Imagine a vocabulary of 6 words and the review "the room was dirty dirty" (after cleaning: "room dirty dirty"):

The output is the vector [0.00, 0.29, 0.82, 0.00, 0.00, 0.00]. "dirty" gets a high score because it appears twice in this review and is rare across the corpus — exactly the kind of word that distinguishes a negative review.

What is the shape of the output?

In this project the vocabulary is capped at 5,000 terms. After processing 9,999 reviews, the TF-IDF output is a sparse matrix of shape:

X = (9,999 rows) × (5,000 columns)

Each row is one review. Each column is one vocabulary word. Most cells are 0 (sparse matrix) because a single review uses only a small fraction of all 5,000 words.

How does this matrix feed into the model?

Each row of the matrix (one review = one 5,000-dimensional vector) becomes a training example. The corresponding label (0 or 1) is the target. Logistic Regression learns a weight for each of the 5,000 dimensions that best separates positive from negative reviews.

What TF-IDF cannot do

TF-IDF is a bag-of-words model — it counts words but completely ignores word order. "Not good" and "Good" could receive similar vectors if "not" is filtered as a stopword. This is a fundamental limitation addressed in Section 7.

05 — Classification algorithm

Logistic Regression — How It Works

Yes, a model IS trained in this project. Logistic Regression is not a rule-based system — it learns the weights from data during the training phase.

The core idea

Logistic Regression calculates a weighted sum of all TF-IDF features, then squashes it through the sigmoid function to produce a probability between 0 and 1.

z = w₀ + w₁·x₁ + w₂·x₂ + ... + w₅₀₀₀·x₅₀₀₀ Dot product of learned weights w with the TF-IDF vector x
P(positive | review) = σ(z) = 1 / (1 + e^(−z)) Sigmoid maps any real number to (0, 1) — a probability
Decision rule

If σ(z) ≥ 0.5 → predict positive (1)
If σ(z) < 0.5 → predict negative (0)

What "training" means

Finding the weights w₁…w₅₀₀₀ that minimise the log-loss (cross-entropy) on the training data. This is done via gradient descent / L-BFGS.

The 80/20 Train-Test Split

The dataset is randomly split: ~7,999 reviews for training (model sees labels and learns weights) and ~2,000 reviews for testing (model never sees these labels — used only to measure real-world performance). The fixed seed random_state=42 ensures the split is reproducible.

Why do we need a test set at all?

To detect overfitting. A model can memorise training data and score 100% on it while failing on new data. The test set simulates unseen real-world reviews and gives an honest estimate of generalisation performance.

The 85% accuracy trap

Because 73% of reviews are positive, a naive model that predicts everything as positive would already score 73% accuracy. The low negative recall (0.56) is the real story — the model misses nearly half of all negative reviews. This is caused by class imbalance.

07 — Business output

Hotel Rating Aggregation

Once the model is trained and validated, it is applied to all 9,999 reviews (not just the test set) to generate a sentiment prediction for each review. Then:

overall_rating(hotel h) = mean( predicted_sentiment ) for all reviews of hotel h = proportion of reviews predicted as positive (1) for that hotel

A hotel with an overall_rating of 1.0 means every single review was predicted positive. A rating of 0.6 means 60% of reviews were positive. The output is then sorted descending to produce a ranked leaderboard.

Input–Output summary of the full system
  • Input sentence → preprocess → TF-IDF vector → logistic regression → P(positive) → binary prediction (0 or 1)
  • Many sentences for one hotel → mean of predictions → hotel score (0–1)
  • All hotels → sorted table + visualisations

08 — Critical analysis

Limitations & How to Fix Them

01
Class Imbalance Skew
73% of reviews are positive. The model sees far fewer negative examples during training, so it develops a bias toward predicting "positive". Negative recall is only 0.56 — the model misses 44% of complaints.
How to fix
(a) Oversampling — Duplicate or synthetically generate negative examples (SMOTE — Synthetic Minority Oversampling Technique).
(b) Undersampling — Randomly remove positive examples to balance classes.
(c) Class weights — Set class_weight='balanced' in sklearn's LogisticRegression to penalise misclassifying the minority class more heavily.
02
Context-Blindness (Bag-of-Words Constraint)
TF-IDF discards word order completely. "not good" and "good, not bad" may have similar vector representations. Sarcasm, irony, and multi-word phrases are invisible to the model.
How to fix
(a) N-grams — Extend TF-IDF to include bigrams/trigrams (e.g., "not_good" as a single feature) via ngram_range=(1,2).
(b) Word embeddings — Replace TF-IDF with pre-trained word2vec or GloVe vectors that encode semantic similarity.
(c) Transformer models — Use BERT or RoBERTa which process the full sentence with self-attention, capturing context, negation, and sarcasm.
03
Vocabulary Feature Capping (top 5,000 words)
Low-frequency but highly diagnostic words (e.g., "mould", "cockroach", "plumbing") may be excluded, causing the model to miss specific complaint categories important for hospitality diagnostics.
How to fix
(a) Increase max_features — Try 10,000 or 20,000 and monitor the accuracy-vs-computation tradeoff.
(b) Use sublinear TF scaling — sublinear_tf=True in TfidfVectorizer reduces the dominance of very frequent terms, giving rarer words more relative weight.
(c) Domain-specific vocabulary — Add a custom vocabulary of known hospitality complaint terms.
04
Linear Modeling Constraints
Logistic Regression assumes sentiment can be separated by a linear boundary in TF-IDF space. Human language is non-linear — sentiment depends on complex interactions between words that a linear model cannot capture.
How to fix
(a) Non-linear classifiers — Use Random Forest, Gradient Boosting (XGBoost), or SVM with an RBF kernel on the same TF-IDF features.
(b) LSTM / GRU — Recurrent networks maintain sequential memory, capturing long-range dependencies in text.
(c) Transformers (BERT) — The current state of the art for NLP; bidirectional attention captures full sentence context. Fine-tuning on hotel reviews would dramatically increase recall on negative examples.
09 — Exam preparation

Professor's Questions — Answered

Q Do you actually train a model in this project, or just use a pre-built one? +

Yes, a model is trained from scratch on this dataset. Logistic Regression starts with random (or zero) weights and iteratively adjusts them using gradient descent to minimise the binary cross-entropy loss on the 80% training split. After training, the learned weights are specific to this corpus — they encode which words in hotel reviews are most predictive of positive or negative sentiment.

Q What exactly happens when you put a sentence into TF-IDF? +

The sentence is first preprocessed (lowercased, punctuation removed, stopwords removed, lemmatized). Then each remaining word is looked up in the pre-fitted vocabulary (5,000 words). The output is a sparse numeric vector of length 5,000 where:

  • Position i holds the TF-IDF score of the i-th vocabulary word in this review.
  • Most positions are 0 (that word didn't appear).
  • A high score at position i means word i is both frequent in this review AND rare across the corpus — a strong discriminative signal.

Critically, the vocabulary and IDF values are fitted on the training set only and then applied (transformed) to new data. This prevents information leakage from the test set.

Q How does the TF-IDF vector get used inside Logistic Regression? +

Logistic Regression computes a weighted dot product between the TF-IDF vector (x) and the learned weight vector (w):

z = w · x = w₁x₁ + w₂x₂ + ... + w₅₀₀₀x₅₀₀₀

This scalar z is then passed through the sigmoid: σ(z) = 1/(1+e^{-z}), yielding the probability of the review being positive. Words with large positive weights push z up (towards positive). Words with large negative weights push z down (towards negative). Most words have weights near zero (irrelevant).

Q Why is accuracy 85% but the model still performs poorly on negative reviews? +

Because of class imbalance. The dataset has 73% positive reviews. A model that predicted "positive" for every single review would achieve 73% accuracy without learning anything. The model here scores 85%, which sounds good, but the negative recall of 0.56 reveals that it misses 44% of genuine complaints.

In a hospitality business context this is dangerous: misclassifying a complaint as positive means the hotel never acts on it. Macro F1 (0.79) is a better summary metric because it averages across classes equally, penalising the poor negative recall.

Q Why is the threshold for "positive" set at rating ≥ 4, and not ≥ 3 or ≥ 5? +

This is a design choice. A 4-star rating clearly indicates satisfaction; a 5-star threshold would create too few positive examples (only top enthusiasts), and a 3-star threshold treats neutral reviews as positive, which blurs the signal. The ≥4 boundary creates a reasonably clean, noise-resistant separation that aligns with real-world hospitality conventions (4–5 = good, 1–3 = problematic).

The tradeoff is that it exacerbates class imbalance — 4-star and 5-star reviews dominate most hotel datasets.

Q Why are we using the star rating at all if the model learns from text? +

The star rating serves as the ground-truth label (y) during supervised learning. We assume the customer who wrote the review and gave 1 star was genuinely negative. This label supervision is what allows the model to learn which text patterns correlate with positive vs negative sentiment.

At inference time (predicting on new reviews), no star rating is needed — the model predicts from text alone. This is the business value: automating sentiment extraction for platforms where reviews arrive without explicit ratings, or to detect mismatches between rating and text.

Q What does "sparse matrix" mean and why does it matter? +

The TF-IDF output is a matrix with 9,999 rows × 5,000 columns = ~50 million values. However, a single review typically uses only 50–200 unique words out of the 5,000 vocabulary — so ~96% of values are zero. Storing all 50 million floats would waste memory. Scipy's sparse matrix format stores only the non-zero positions and values, reducing memory usage by over 95%. Logistic Regression in scikit-learn is optimised to work directly with sparse inputs.

Q Can this model understand "The hotel was NOT great"? Why or why not? +

Poorly, or not at all. After stopword removal, "not" may be filtered out. Even if kept, TF-IDF creates two separate features: "not" and "great". The model only sees the weights of each word independently — it has no way of knowing that "not" negates "great". The word "great" has a strong positive weight from training, so the review might actually be classified as positive — a false prediction.

Fixes: n-gram TF-IDF (treating "not_great" as one feature), or a Transformer model that uses attention to process the full sentence context.

Q What is the difference between precision and recall, and which matters more here? +

Precision = of all reviews I called negative, how many really were negative? (False positive cost)
Recall = of all reviews that were genuinely negative, how many did I catch? (False negative cost)

In this business context, recall on the negative class matters more. Missing a genuine complaint (false negative) means the hotel doesn't know about a real problem and can't fix it — a costly business error. A false positive (calling a happy guest "unhappy") triggers unnecessary follow-up but is less harmful. So maximising negative recall, even at some precision cost, is the right objective — which this model currently fails at (0.56).

Q How would you improve this project if you had more time? +

Short-term improvements (same model family):

  • Add bigrams/trigrams to TF-IDF to capture negation patterns
  • Use class_weight='balanced' to address imbalance
  • Increase vocabulary to 10,000+ features
  • Tune the regularisation parameter C in Logistic Regression via cross-validation

Long-term improvements (better architectures):

  • Fine-tune a pre-trained BERT model on this dataset — contextual embeddings would solve negation/sarcasm issues
  • Use aspect-based sentiment analysis to score specific hotel attributes (cleanliness, staff, location) separately
  • Collect more negative reviews (better data collection strategy) or use data augmentation
Q Why fix random_state=42 during train-test split? +

Reproducibility. Without a fixed seed, every run of the code produces a different random split, which means reported accuracy numbers could vary from run to run — making it impossible to know if a change you made actually improved the model, or if you just got a lucky split. Fixing the seed ensures every run of the code produces exactly the same training and test sets, so results can be independently verified and compared.

Q What is the IDF component for, and what would happen without it? +

IDF (Inverse Document Frequency) down-weights words that appear in almost every document. Without IDF, raw Term Frequency (TF) would heavily weight words like "hotel", "room", "stay" — words that appear in nearly every review and carry zero discriminative power for positive/negative classification. IDF divides by the number of documents containing the term (log-scaled), so ubiquitous words get scores near zero regardless of how often they appear in a single review. This ensures the vector is dominated by distinctive words rather than common filler words.
